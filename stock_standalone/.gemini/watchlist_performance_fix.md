# Watchlist 性能优化修复报告

**日期**: 2026-02-11  
**问题**: 打开 Watch 标签页后,`update_watchlist` 持续高频调用且耗时严重 (900ms - 3300ms)

## 🔴 问题诊断

### 根本原因
在 `_update_watchlist_queue()` 函数中,**每次刷新时对每只股票都执行了重复的昂贵操作**:

```python
# ❌ 原代码 - 在循环内重复查询
for i, row in enumerate(df.itertuples()):
    # 每只股票都执行:
    main_window = self._find_main_window()  # 查找主窗口
    hot_concepts = self._get_recent_hot_concepts(days=2)  # 查询数据库
    # 板块匹配逻辑...
```

### 性能影响
- **时间复杂度**: O(n) → 每只股票都重复执行相同查询
- **实际耗时**: 
  - 观察池 50 只股票 → 50 次主窗口查找 + 50 次数据库查询
  - 观察池 100 只股票 → 100 次主窗口查找 + 100 次数据库查询
- **日志表现**: 
  - 正常: 100ms 以内
  - 异常: 900ms - 3300ms (超标 9-33 倍)

### 触发场景
1. **全量重建路径** (第 974-1054 行): 对每只股票都查询主窗口和热榜
2. **增量更新路径** (第 894-912 行): 板块为空时也会触发查询

## ✅ 解决方案

### 核心优化策略
**完全移除不必要的查询操作** - 板块信息只在用户点击时才需要,不应在刷新时计算

### 具体修改

#### 1. 移除循环内的重复查询
```python
# ✅ 优化后 - 直接使用数据库字段
for i, row in enumerate(df.itertuples()):
    # 板块信息直接使用数据库字段,不做额外查询
    sector = str(row.sector) if row.sector else ""
    if sector and len(sector) > 20:
        sector = sector[:20]
```

#### 2. 简化增量更新逻辑
```python
# ✅ 移除了板块信息增量校准逻辑
# 原来的 894-912 行板块匹配代码已删除
```

#### 3. 修复 Lint 警告
```python
# ✅ 处理未使用的返回值
if is_sorted:
    _ = self.watchlist_table.setSortingEnabled(False)
```

## 📊 预期性能提升

### 时间复杂度优化
- **优化前**: O(n × m) 
  - n = 股票数量
  - m = 主窗口查找 + 数据库查询 + 板块匹配
- **优化后**: O(n)
  - 仅遍历股票,无额外查询

### 预期耗时
- **观察池 50 只股票**: 900ms → **< 50ms** (提升 18 倍)
- **观察池 100 只股票**: 3300ms → **< 100ms** (提升 33 倍)

## 🎯 设计原则

### 职责分离
- **刷新逻辑**: 仅负责显示数据库中已有的数据
- **点击交互**: 需要额外信息时才查询 (如 `_find_main_window()`)

### 数据源策略
- **板块信息**: 使用数据库 `sector` 字段 (由数据采集时填充)
- **实时价格**: 使用缓存 `_last_price_map`
- **主窗口数据**: 仅在用户交互时访问
