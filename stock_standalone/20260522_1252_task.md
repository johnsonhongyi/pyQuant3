# 2026-05-22 12:52 板块聚合全量与分层异步解耦结构设计与优化清单 (Layered Async Architecture & Decoupled Post-Aggregation UI Notification)

## 1. 痛点分析与深入定位

### 🔴 核心痛点
虽然我们在 CPU 层面通过 Early-Exit 过滤和 Single-pass Concept Cache 极速缩短了 `_aggregate_sectors` 内部板块大循环的计算耗时，但从系统全局的并发架构来看，原有的打分与板块聚合设计依然存在以下问题：
1. **未解耦的同步调用链 (Synchronous Callback Block)**：在个股打分分片完成的瞬间，`_finish_score` 在当前分片计算线程中同步调用了 `_aggregate_sectors`。哪怕是增量，由于计算任务仍然卡在这个线程中，导致个股打分状态机无法亚毫秒级瞬间关闭，且由于 Python GIL 锁的存在，重度行情下密集执行 CPU 计算依然会导致主 GUI 线程的卡死与 lag。
2. **打分完成与 UI 更新的同步冲突 (Signal & Thread Contention)**：在 `_finish_score` 内部执行完聚合后，它同步触发了 `self.on_score_finished()` 以通知外部 UI 进行刷新。若此时主 GUI 线程与打分线程因为某些内存数据而发生隐性的排它锁锁竞争，就会造成极其一致的 3-5 秒界面整体卡死假象。
3. **高频行情下的无效板块重算 (Inefficient Redundant Computations)**：当多路行情爆发、打分任务极其频密地推来时，系统会为每一轮打分结果都同步启动板块聚合。但在实际分析中，我们只需要呈现**最新的一张板块快照**，之前中间未及呈现的版本属于完全无意义的重复算力开销。

## 2. 分层异步重构方案与“完成后更新”模式

针对这一痛点，我们彻底重构了数据引擎，实施了**分层异步解耦（Layered Asynchronous Decoupling）**的极客级优化：

### 🛠️ 核心架构一：异步板块工作线程 (Sector Aggregation Worker Thread) 异步处理
- **高可靠异步聚合队列**：在 `BiddingMomentumDetector` 类中引入了独立的异步板块聚合队列 `self._async_sector_agg_queue = Queue()` 以及退出判定事件 `self._stop_event = threading.Event()`。
- **常驻守护工作线程**：在 `_start_register_workers` 中专门启动了常驻守护工作线程 `async_sector_agg_worker`。该线程作为板块计算的专属引擎，独立、安全地运行在后台，完全移除了打分线程和 GUI 线程的所有同步执行开销。
- **打分收尾 0.00ms 物理零阻塞**：现在个股打分状态机跑完后，在 `_finish_score` 内部仅将变动个股（`dirty_codes`）作为任务极速 `put()` 入队列，**瞬间结束退出**，耗时在 CPU 亚微秒级，完美释放打分状态机！

### 🛠️ 核心架构二：防抖合并折叠高时效聚合 (Coalesced Queue Debouncing)
- **多任务智能折叠**：在后台工作线程中，当获取到第一个任务后，程序会自动扫描并取出队列里当前积压的所有待聚合任务。
- **参数动态排重与升级**：将积压任务的 `active_codes` 进行 `set.update` 合并，若有任一全量刷新任务（`None`），则直接折叠升级为全量聚合任务，最终合并为**单次最具时效性、最新的聚合重构**，从根本上过滤了高频行情下的无效计算积压，保护 CPU 趋零波动。

### 🛠️ 核心架构三：“完成后更新”异步模式 (Update-After-Complete Pattern)
- **数据完全一致再投递**：将 `self.on_score_finished()` 这一触发外部 GUI 刷新的核心回调，从打分线程剥离，并**完美移入到异步板块工作线程在执行完 `_aggregate_sectors` 之后**。
- **极致的线程亲和与流畅度**：打分和板块聚合均在后台计算完毕、内存结果 `self.active_sectors` 彻底沉淀一致后，才从后台异步抛出信号，UI 接收到信号后执行亚毫秒级的刷新重绘。彻底消除了由于“边计算、边通信”引发的锁竞争和 UI 卡死假死！

### 🛠️ 核心架构四：退出的安全生命周期加固 (Graceful Exit Protection)
- **无残留哨兵投递**：在 `stop()` 退出释放函数中，除了设置 `_stop_event.set()`，更对 `_async_sector_agg_queue`、`_subscribe_queue` 及 `_sector_update_queue` 自动投递 `None` 退出哨兵，确保各后台 Daemon 线程能干净、利落地退出，彻底消除了退出阶段临时目录（_MEI）被文件描述符强占的顽疾！
