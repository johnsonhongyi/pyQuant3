# 2026-05-19 20:45 - 彻底解决 Nuitka 环境下 15:30 自动收盘 UI 饥饿与死锁问题

## 1. 背景与痛点
在 Nuitka 独立打包编译运行环境下，当系统运行至 15:30 触发自动收盘落库归档任务时，Tkinter 主界面会发生假死/挂起，Watchdog 报出 UI 阻塞警告。
分析其原因在于：
1. **GIL 被“长期独占”模型**：`run_15_30_task` 启动 `SectorBiddingPanel` 面板实例并触发后台评分计算。由于 15:30 是全量行情（5500+ 只股票），后台 `DataProcessWorker` 子线程启动了极高密度的 C-level 大计算，迅速占满 CPU 且牢牢控制住 GIL。
2. **写日志与 initialization 级联死锁**：在主线程 `__init__` 返回的临界点上，系统过早地调用了 `on_realtime_data_arrived()` 往子线程喂数。主线程在写日志（`logger.info("📡 [SectorPanel] Cold start initialized...")`）以及返回至 `wait_q.put(True)` 之前，被迅速苏醒开始大计算的子线程夺去 GIL 导致饿死，无法向 `wait_q` 写入数据，从而导致 `run_15_30_task` 发生 10 秒超时以及 UI 彻底假死。
3. **子线程同步重复计算**：在前面的计算队列完成后，子线程 `run_15_30_task` 居然同步调用了 `detector.update_scores()` 进行第二次全量评分计算。

## 2. 核心优化方案与实施步骤
针对上述问题，我们实施了极具智慧的“降维打击”与“自愈防线”：

### Step 1: 延迟冷启动数据喂入，保障主线程瞬间安全返回
- **修改文件**：[sector_bidding_panel.py](file:///d:/MacTools/WorkFile/WorkSpace/pyQuant3/stock_standalone/sector_bidding_panel.py)
- **改动内容**：将 `__init__` 构造函数末尾的 `self.on_realtime_data_arrived(self.main_window.df_all, force_update=True)` 行情喂数动作移除。
- **物理机制**：主线程在初始化时不需要急着向后台喂数。在 `showEvent` 里面已经有 `QTimer.singleShot(500, self.manual_refresh)` 逻辑，该定时器在 500ms 后会自动调用 `manual_refresh`，而 `manual_refresh` 内会自动检查 `main_window.df_all` 并安全喂数。因此移除此处的同步喂数，不仅主线程能以 0 微秒延迟瞬间初始化并成功向 `wait_q` 投递 `True`，同时彻底断绝了子线程大计算在主线程构造函数返回前夺取 GIL 导致饿死死锁的根本病灶！

### Step 2: 清除子线程内不安全的 UI 控件修改
- **修改文件**：[sector_bidding_panel.py](file:///d:/MacTools/WorkFile/WorkSpace/pyQuant3/stock_standalone/sector_bidding_panel.py)
- **改动内容**：在异步回调 `_on_detector_ready` 里面，去除了直接跨线程操作 UI 控件 `self.status_bar.showMessage` 的代码，仅保留线程安全的 `self._force_update_requested = True`。这确保了在 Nuitka C 独立运行时的 100% 线程安全。

### Step 3: EOD 任务跳过个股重算，仅执行极速板块聚合
- **修改文件**：[instock_MonitorTK.py](file:///d:/MacTools/WorkFile/WorkSpace/pyQuant3/stock_standalone/instock_MonitorTK.py)
- **改动内容**：将 `run_15_30_task` 子线程中的 `detector.update_scores()` 改为 `detector.update_scores(skip_evaluate=True)`。
- **物理机制**：由于 `SYNC-DENSITY-CHECK` 循环已经确保了后台工作线程将 5500+ 个股的分数全部算好（"数据已熟透"），子线程在收盘落库时根本没有必要再次从零将个股计算一遍。指定 `skip_evaluate=True` 可以使系统在一微秒内仅执行极速的板块聚合分计算，彻底释放了 CPU 与 GIL，打通了极速 EOD 自动归档通道！

### Step 4: 修复 Nuitka 批处理脚本语法隐患
- **修改文件**：[nuitka_build_console.bat](file:///d:/MacTools/WorkFile/WorkSpace/pyQuant3/stock_standalone/nuitka_build_console.bat)
- **改动内容**：补齐了第 198 行 `--include-module=signal_bus` 末尾遗漏的 `^` 续行符。这确保了批处理多行命令的连贯性，防止编译选项发生语法断裂。
