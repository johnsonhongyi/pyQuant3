## 2. 痛点剖析：为什么总是在“大涨后”才看到？

在实盘中，交易员经常遇到“筛选出几百只票，但看过去时很多票已经拉了 8% 甚至涨停”的痛点。这主要由以下原因造成：
1. **排序权重失衡**：默认的实时监控看板通常按照 `percent`（日内涨幅）或 `buyscore`（买入评分）进行降序排列。而 `buyscore` 本身与涨幅高度正相关，这导致涨幅最大的票被顶在最上方，吸走了交易员的注意力。
2. **缺乏二次空间过滤**：没有将“当前涨幅”和“均线偏离度”加入强过滤逻辑，导致刚刚从支撑位起步、利润空间最大的个股，被埋没在几百只股票的代码海洋中。

## 3. 黄金突破方案：“金子结构”起爆低吸过滤漏斗

为了在**“大涨前”**、在**“水下或刚红盘”**时就精准逮住最强个股，我们在 pyQuant3 原有 SWS 逻辑上，叠加了一套**「金子结构」二次漏斗**模型：

| 核心维度 | 指标参数 | 策略设计原理 |
| :--- | :--- | :--- |
| **位置安全度** | `0.5% <= percent <= 3.5%` | 股价刚刚翻红或小幅拉升，**尚未大涨**，留足利润空间。 |
| **开盘安全度** | `-2.5% <= open_pct <= 1.2%` | 水下洗盘或微高开抢筹，排除过度高开的追高风险。 |
| **均线支撑度** | `0% <= dist_ma5 <= 4.0%` | 当前价距离 5日线 极近，说明刚踩稳 5日均线 支撑拐头向上。 |
| **主力吸筹度** | `dff2 >= 15` 且 `dff3 >= 80` | **最核心条件**。大单持续爆量流入，说明主力在低位疯狂吸筹。 |
| **量能健康度** | `0.8 <= vol_ratio <= 2.2` | 量比温和放大，排除缩量诱多和爆量滞涨。 |

## 5. 实盘监控系统优化与工程化落地

### 5.1 在 `stock_live_strategy.py` 中增加低吸策略过滤器
在实盘扫描或 CSV 导出逻辑中，增加一个定制的 `get_golden_sws_lowbuy` 方法：
```python
def filter_golden_structure(df_all):
    # 基础过滤: SWS低开高走策略输出的代码池
    # 叠加“金子结构”起爆漏斗
    df_filtered = df_all[
        (df_all['open'] / df_all['lastp1d'] >= 0.975) & (df_all['open'] / df_all['lastp1d'] <= 1.012) & # -2.5% 到 1.2% 开盘
        (df_all['percent'] >= 0.5) & (df_all['percent'] <= 3.5) &                                    # 0.5% 到 3.5% 涨幅
        (df_all['close'] / df_all['ma5d'] >= 1.0) & (df_all['close'] / df_all['ma5d'] <= 1.04) &        # 距离5日线极近
        (df_all['dff2'] >= 15.0) & (df_all['dff3'] >= 80.0) &                                        # 主力大单暴量流入
        (df_all['vol_ratio'] >= 0.8) & (df_all['vol_ratio'] <= 2.2)                                  # 温和量比
    ]
    return df_filtered
```
