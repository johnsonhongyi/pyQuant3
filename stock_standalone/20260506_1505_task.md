# 2026-05-06 15:05 修复语音播报 SAPI5 内存不足与未注册类崩溃

## 任务背景 (Background)
用户报告在盘中信号密集爆发（如 `14:53:08` 形态信号多只触发）的高负载情况下，`trade_visualizer_qt6.py` 的语音播报进程抛出以下致命错误日志并可能导致语音服务卡死或失效：
```text
[05-06 14:54:36] ERROR:trade_visualizer_qt6.py(_voice_worker:280): [VoiceProcess] Engine error: (-2147024882, '内存资源不足，无法完成此操作。', ('没有注册类\r\n', None, None, 0, None))
```
其中，`-2147024882` 为 Windows HRESULT 错误码 `0x8007000E` (`E_OUTOFMEMORY`，即内存/资源不足)；而 `'没有注册类\r\n'` 对应 `0x80040154` (`REGDB_E_CLASSNOTREG`)。

## 根因剖析 (Root Cause Analysis)
经对 `trade_visualizer_qt6.py` 的 `_voice_worker` 语音播放工作循环深入排查，发现了导致该 Windows COM/SAPI 资源耗尽的关键逻辑缺陷：
1. **循环内高频初始化与反初始化**：原代码在 `while stop_flag.value:` 循环内部，**每接收一条播报文本**，都会强行执行一次 `pythoncom.CoInitialize()`、清理缓存并执行 `pyttsx3.init()`，播报完后立刻在 `finally` 中调用 `engine.stop()`, `del engine` 并在最后调用 `pythoncom.CoUninitialize()`。
2. ** apartment 模型崩塌与句柄泄漏**：在信号爆发期，短时间内会连续有十几条消息入队，导致在同一个线程内以毫秒级频率极高频地执行 `CoInitialize` / `CoUninitialize`。这会直接导致 Windows COM 套间引用计数错位，且 `pyttsx3.init()` 重建底层 SAPI 引擎（需加载外部 DLL、大量访问注册表并分配 COM 结构）极其沉重。高频反复创建销毁最终触发 Windows SAPI5 注册表句柄资源耗尽，抛出“内存资源不足”或“没有注册类”。

## 优化与自愈方案 (Solutions)
为了根治这一顽疾，我们参考了量化系统底层的工业级稳定性标准以及 `alert_manager.py` 的高可靠实践，重构了 `_voice_worker` 的生命周期：

### 1. 提升 COM 套间生命周期至线程级 (Thread-level COM Apartment)
将 `pythoncom.CoInitialize()` 和 `pythoncom.CoUninitialize()` 移出高频消息循环：
- 线程启动时，在进入 `while stop_flag.value` 循环前调用 **一次** `CoInitialize`。
- 线程完全退出、释放所有资源时，调用 **一次** `CoUninitialize`。
- 完美契合 Windows 官方对 COM 套间生命周期的管理规范，从源头上切断了 COM Apartment 崩塌和引用计数错乱导致的异常。

### 2. 隔离并重新实例化语音引擎 (Loop-level pyttsx3 Re-instantiation)
- **发现**：Windows SAPI5 的 pyttsx3 引擎如果作为全局单例复用，当单次 `runAndWait()` 结束后，其内部的系统级事件泵已被断开。如果在没有重建引擎实例的情况下强行执行下一次 `say()` 调用，后续的语音播报将会被系统底层静默忽略（即“播报无声音”）。
- **解决方案**：我们在保持 COM 线程级生命周期绑定的同时，将 `pyttsx3.init()` 依然作为独立隔离播报周期的主体保留在循环内部。每条消息出队时实例化一个干净的 `engine`，在 `finally` 块中通过 `engine.stop()`, `del engine` 物理释放它。
- **收益**：既消除了高频 `CoInitialize` 导致的系统资源和套件损坏崩溃，又恢复并保障了每条语音消息 100% 的实盘稳定输出（声音完美发出）。

### 3. 加持自愈式异常重置机制 (Self-healing Re-instantiation)
- 保留高可靠的 try-except 容错机制：当 `engine.say()` 或 `runAndWait()` 遭遇任何 SAPI5/COM 层面的异常时，系统会安全捕获它，并在 `except Exception as e` 分支中主动调用 `engine.stop()`, `del engine` 并将 `engine = None`。
- 这样，下一条消息入队时，系统会自动、无缝、干净地在健康的 COM 套件下重新初始化（`pyttsx3.init()`）一个全新的引擎。这不仅极大地提升了容错率，还保证了 100% 的盘中自愈恢复力。

## 实施状态 (Implementation Status)
- [x] **提升 COM 初始化生命周期**：将 `pythoncom.CoInitialize` 和 `CoUninitialize` 移出消息循环，完美绑定至 `_voice_worker` 线程的最外端，彻底解决了 Apartment 崩塌和 Windows 句柄内存不足。
- [x] **落地隔离式引擎生命周期**：在消息循环内动态实例化 `pyttsx3.init()`，并在每一条消息播放完毕后，于 `finally` 块中显式执行 `engine.stop()` 和 `del engine` 安全解构。这既解决了 SAPI5 状态机单例复用引起的“静默无声音”硬伤，又确保了每一条语音播放完毕后资源被物理、安全地归还给操作系统。
