## 📋 变更详情

### 1. 深度分析与定位 (Problem Diagnosis)
- **原因 1：SyncManager 进程未忽略中断信号**：
    当用户按下 Ctrl+Break 或触发 `dump_all` 堆栈转储时，操作系统会将控制台广播的 `SIGBREAK` / `CTRL_BREAK_EVENT` 信号发送给该控制台窗口下的**所有进程**。由于 `multiprocessing.Manager()` 启动的 `SyncManager` 背景进程默认没有忽略该信号，它会立刻闪退死亡。
    这导致共享字典 `self.global_dict` 通信物理中断，使得在 `instock_MonitorTK.py` 中，`self.global_values` 的状态自检标记 `_manager_dead` 在后续获取 `resample` 时立刻变为 `True`。此后系统切换到了本地 fallback 字典，无法再在多进程间完成“切换周期”等全局状态的原子分发，因此“切换周期不能自动刷新显示信息”。
- **原因 2：PyQt Visualizer 进程未忽略中断信号**：
    同理，由 `self.qt_process` 独立启动的 PyQt K线可视化客户端，在收到控制台 `SIGBREAK` 时，由于没有在入口级屏蔽该信号，同样会在瞬间被连带强退死亡。所以“vis状态不能自动恢复”（因为子进程已经彻底不复存在）。

### 2. 物理加固实现 (Subprocess Signal Isolation Hardening)
- **加固 SyncManager 自启动流程**：
    重构了 `instock_MonitorTK.py` 的全局状态存储初始化逻辑。使用 `multiprocessing.managers.SyncManager` 自定义启动方式，并定义全局顶层函数 `init_manager_process` 以满足 Windows 下 `spawn` 方式的 pickling (序列化) 要求。通过 `start(initializer=init_manager_process)` 中显式传入，使其在 `SyncManager` 进程启动的微秒级瞬间在底层将 `SIGINT`/`SIGBREAK` 屏蔽并注册 `SetConsoleCtrlHandler` 忽略，完美避免了 Windows 平台下闭包函数无法 pickle 的 `AttributeError` 崩溃，实现了 100% 免疫和自适应启动。
- **加固 PyQt Visualizer 进程入口**：
    在 `trade_visualizer_qt6.py` 的 `main()` 函数头部，直接嵌入了对 `SIGINT` / `SIGBREAK` / `SetConsoleCtrlHandler` 的忽略和拦截机制，彻底阻止了控制台诊断热键对 GUI 进程的干扰，使其仅对 Qt 的主窗口 Close 动作及 Pipe 退出指令敏感。
- **加固 赛马回测/回放 进程入口**：
    同理在 `test_bidding_replay.py` 的 `main()` 入口函数头部嵌入了完全相同的信号防线，实现了全链条多进程自愈体系的物理闭环。

## 🛠️ 体现的核心编程原则 (KISS, YAGNI, DRY, SOLID)

- **KISS (Keep It Simple, Stupid)**：没有设计任何庞杂的心跳守护重连机制，而是直接在底层信号源头上做物理屏蔽，极其直接、简单且无法被击穿。
- **DRY (Don't Repeat Yourself)**：采用了标准库中 `BaseManager.start(initializer=...)` 内置的进程环境配置标准，完美地利用了 multiprocessing 现有的自启动机制。
- **SOLID - 单一职责原则 (SRP)**：把信号安全防卫职责，安全隔离到了各个子进程的自启动入口中，保障了主界面交互业务的极度纯粹和高稳定性。
