# Task: 修复赛马回测时间轴秒数跳变与实盘时间退避 (Fixed Replay Timeline Seconds Jitter & Simulation Timestamp Fallback)

> **创建时间**：2026-05-06 15:30  
> **状态**：✅ 已完成  
> **涉及模块**：BiddingMomentumDetector (`bidding_momentum_detector.py`), BiddingRacingRhythmPanel (`bidding_racing_panel.py`)

---

## 🎯 任务背景与核心痛点
1. **时间重置抖动**：在赛马回测录像回放 (Replay / Simulation Mode) 中，当某只股票 K 线的 `ticktime` 数据缺失或尚未对齐时，计算评估层会自动兜底使用 `time.time()` (当前实盘自然物理时间) 作为时钟源，并写入 `last_data_ts` Thread-safe 缓存。
2. **时钟相互打架**：这导致 `self.last_data_ts` 时钟在“模拟回放时间”（例如开盘 09:30 附近）与“实盘自然物理时间”（例如当下的下午/晚上时间）之间高频交叉写入和疯狂跳变。
3. **秒数取模冲突**：UI 端 `update_visuals` 在更新时间轴时强制使用分钟格式化 `strftime("%H:%M:00")`，而回放推送的消息流则使用的是高精度秒级时间（例如 `09:30:45`），导致秒钟部分在 `:00` 和具体秒数之间高频拉锯闪烁，造成了回测录像重播期间时间疯狂变化的极度不佳体验。

---

## 🛠️ 解决方案与核心改动
### 1. 阻断回放/历史模式下的物理时钟污染
在 `bidding_momentum_detector.py` 的两处 `time.time()` 兜底赋值逻辑中，引入了对 `simulation_mode` (仿真回测) 和 `in_history_mode` (历史复盘) 属性的智能安全门限：
- **评估前 K 线缺时兜底** (Line ~2205)：
  ```python
  if data_ts <= 0:
      data_ts = ts_obj.first_breakout_ts if ts_obj.first_breakout_ts > 0 else self.last_data_ts
      if data_ts <= 0:
          if not (getattr(self, 'simulation_mode', False) or getattr(self, 'in_history_mode', False)):
              data_ts = time.time()
  ```
- **评估后 `last_data_ts` 推进兜底** (Line ~2636)：
  ```python
  elif data_ts == 0:
      if not (getattr(self, 'simulation_mode', False) or getattr(self, 'in_history_mode', False)):
          self.last_data_ts = time.time()
  ```
这确保了回放和历史复盘期间的 `self.last_data_ts` 仅接受纯粹的仿真模拟数据时间戳，物理阻断了 `time.time()` 的外部干扰。

### 2. 对齐高精度秒级时间戳渲染
在 `bidding_racing_panel.py` 的 `update_visuals` 主渲染周期内，将时间轴进度时钟的时间格式化改用高精度秒级对齐：
```diff
- t_str = datetime.datetime.fromtimestamp(curr_time).strftime("%H:%M:00")
+ t_str = datetime.datetime.fromtimestamp(curr_time).strftime("%H:%M:%S")
```
这彻底打通并消除了分钟取整导致的 `:00` 和实际回放秒数（如 `:45`）之间的冲突。

### 3. 保障正式实盘绝对不受任何副作用波及
所有改动均经过精准门控包裹，只针对 `simulation_mode` 和 `in_history_mode` 进行逻辑优化，实盘数据管道与时钟完全未被修改，100% 保持原本完美的运行状态。
