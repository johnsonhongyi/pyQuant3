# 20260507_1558 任务清单：修复启动 Python 后快捷键无反馈与 SIGBREAK 按键不响应

## 🎯 任务目标
解决用户在启动系统后，按下的两个排查/转储快捷键（Ctrl+Alt+D 或者是 SIGBREAK 对应的 Ctrl+Break）没有任何物理及终端反馈的体验硬伤，实现 100% 可感知的物理反馈、系统级全局热键以及底层 OS 级控制台回调的双重备份机制。

---

## 📌 完成情况
1. **重构并增强顶级堆栈诊断方法 `dump_all()`**：
    - **独立存盘**：支持将当前的线程堆栈（Traceback）直接物理转储存盘至 `instock_dump.log`。即使控制台完全不可见、或者由于无前台控制台终端（如 pythonw 启动）导致控制台无输出，用户也可以直接通过此文件查阅完整的线程状态，极具生产环境实用性。
    - **日志对齐**：事件触发时自动向 `instock_tk.log` 中记录一条 `logger.warning` 诊断信号触发提示，为问题排查提供清晰的可追溯记录。
    - **极致物理反馈**：在 `dump_all()` 底层启动一个独立的守护线程，强力调用 Windows 原生 `ctypes.windll.user32.MessageBoxW` 弹出置顶物理对话框。即便 Tkinter 或 PyQt6 已经完全陷入 GIL 竞争死锁，得益于独立守护线程与 OS 系统窗口机制，此原生弹窗依然能够毫秒级弹出提示用户 “转储已成功”。
2. **底层 OS 线程级硬核拦截：彻底解决 `signal.signal(signal.SIGBREAK, ...)` 在主线程死锁卡死时不响应的问题**：
    - 在 Python 中，传统的 `signal.signal` 所注册的处理函数由于语言机制的硬性约束，只能在主线程（Main Thread）中被解释器调度执行。因此，当主线程发生死锁（例如进入死循环、遭遇加锁死锁、或由于 heavy operation 长期独占 GIL）时，传统的 Ctrl+Break 触发的 `SIGBREAK` 信号处理 Lambda 函数将**完全无法执行**，造成“按键无反应”的假象。
    - **破局方案**：我们在 `instock_MonitorTK.py` 中引入了 Windows 控制台底层 C 接口 `SetConsoleCtrlHandler`，直接向 OS 内核注册 C-style 的控制台控制回调函数 `win_console_ctrl_handler`。
    - **100% 独立响应**：一旦用户按下 `Ctrl+Break`（即触发 `CTRL_BREAK_EVENT`），Windows 内核会自动唤醒一个全新的、完全独立的操作系统级控制台线程（Console Thread）同步执行此回调。它不仅完全独立于 Tkinter/PyQt6 GUI 主线程上下文，甚至完全绕过了 Python 主线程的事件循环。即使主线程彻底卡死死锁，该系统控制台线程也必定能瞬间、毫无滞后地呼出 `dump_all()`，完成堆栈的安全落盘和 native MessageBox 弹窗。
3. **将 `Ctrl+Alt+D` 强力纳入 Win32 `RegisterHotKey` 全局热键系统**：
    - 原本 `Ctrl+Alt+D` 热键只通过 `keyboard` 模块低级钩子（Low-level Hook）注册，在 Windows 系统高防、多任务、管理员权限差异以及程序长时间运行后极易被 OS 自动卸载，从而彻底失灵。
    - 我们将 `Ctrl+Alt+D` (虚拟键码 `0x44`，控制/阿尔特组合键) 作为 `Offset 8` 强力注册为系统级全局热键（通过 `setup_global_hotkey` 实现）。
    - 回调直接调用顶级函数 `dump_all`，完美绕过了重入 GUI 主线程的 `_schedule_after` 逻辑，即使主线程由于死锁假死，系统级热键也能完美响应。
4. **补齐 Tkinter 本地窗口快捷键降级模式**：
    - 当 `setup_global_hotkey` 降级到 `LOCAL` 本地窗口模式时，自动判断 `desc` 字段中的 Ctrl/Alt 修饰符，完美将 `"Ctrl+Alt+D"` 转换为合法的 Tkinter 键盘绑定形式 `"<Control-Alt-d>"` 进行拦截。
