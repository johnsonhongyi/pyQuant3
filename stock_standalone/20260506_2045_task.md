## 🎯 任务背景与核心痛点
1. **Windows 致命崩溃 (0xc0000096)**：在关闭或手动退出赛马回测/实盘监控进程时，有一定概率在控制台报出 `Windows fatal exception: code 0xc0000096`（特权指令违规，或伴随 `0xc0000005` 内存访问冲突）。
2. **信号回调僵尸指针**：当主 GUI 窗口关闭并触发 `on_panel_closed()` 流程时，后台的工作线程（如 `ReplayWorker` 或 `LiveWorker`）仍可能在微秒级的延迟下，通过 `progress_update.emit()` 或 `status_update.emit()` 尝试投递事件到已被销毁的主窗口组件上。
3. **出序垃圾回收 (GC Teardown Race)**：在 Python 进程退出并销毁模块时，Python 的垃圾回收器会以不确定的顺序销毁 Qt/C++ GUI 对象，而此时底层的 native OS 线程仍处于最后一步的清理注册期。对已经释放的 C++ 资源/指针的并发引用，会导致 Windows 操作系统直接发出特权指令或内存访问冲突的中断。
4. **退出假死与响应迟滞**：当点击退出时，若后台计算线程正处于耗时较大的数据接收、板块跑分周期中（单次循环可能耗时几百毫秒甚至数秒），界面在无期限的 `worker.wait()` 等待中产生严重的卡住与挂起现象，无法第一时间响应关闭指令。

---

## 🛠️ 解决方案与核心改动

### 1. 实施前置信号断开保护 (Pre-emptive Signal Disconnection)
在 Live 模式和 Replay 模式下的主窗口关闭连接入口（`on_panel_closed`），以及 `app.exec()` 之后的物理终点，均引入了对 `worker` 信号的高效 `disconnect()` 断开保护：
- **Live 模式信号断开**：
  ```python
  try:
      worker.progress_update.disconnect()
  except: pass
  try:
      worker.status_update.disconnect()
  except: pass
  ```
- **Replay 模式信号断开**：
  ```python
  try:
      worker.progress_update.disconnect()
  except: pass
  ```
这物理阻断了工作线程退出期间任何残留通知的跨线程派发，使已被销毁的 UI 界面免受任何“幽灵回调”的冲击。

### 2. 打通 `run_replay` 数据循环快速退出开关 (Fast-Exit Breakpoint)
在 [run_replay](file:///d:/MacTools/WorkFile/WorkSpace/pyQuant3/stock_standalone/test_bidding_replay.py#L396) 数据处理热点循环的 `publisher.update_batch` 重载步骤之前，强制插入了快速退出判定：
```python
if ui_callback and not ui_callback(None): break
```
当用户点击退出并将 `self.is_running` 设为 `False` 时，即便当前个股或板块跑分运算产生数秒的耗时突刺，也会在下一次数据泵入执行前被立即掐断，瞬间跳出循环，使线程进入回收流程。

### 3. 优化线程回收等待时限 (Wait Timeout Throttling)
在 Live 和 Replay 模式下的窗口关闭逻辑以及 `app.exec()` 退出点，将原先可能导致界面严重挂起假死的无限制阻塞 `worker.wait()` 和超大时限 `worker.wait(1000)` 替换为高敏感的 10ms 防抖超时门槛：
```python
worker.wait(10)
```
这在保持线程对象生命周期自然终结的同时，使点击退出后的界面等待响应降至亚毫秒级，不再无意义地在 CPU 热循环深处发生假死。

### 4. 引入隔离进程物理退出 (`os._exit(0)`)
在主窗口完全关闭、所有线程完美 `stop()` 以及所有持有的子进程（`active_children()`）被强制回收之后，调用工业级高可用退出原语 `os._exit(0)`。强力绕过了 Python GC 释放 C++ GUI 对象时由于销毁顺序错乱和未完成终结的 native 线程指针并发争用而引发的崩溃，确保了独立的赛马/回测子进程完美、静默且瞬间干净利落地退出。
