# Task: 实现强退 Ctrl+C 与紧急子进程回收机制 & 修复友好窗口名称丢失
日期时间: 2026-05-23 14:05

## 1. 目标与痛点
- **痛点 A (僵尸残留)**: 在多进程架构下，操盘手经常使用 `Ctrl+C` 强退（或者程序异常崩溃时触发），导致后台 `trade_visualizer_qt6.py` (可视化子进程) 和 `hotkey_rotator.py` (热键子进程) 无法即时释放，成为孤儿/僵尸进程，占用 Named Pipe (`\\.\pipe\instock_tk_pipe`) 端口，影响下一次系统正常启动。
- **痛点 B (名称丢失)**: 在多进程解耦以后，新拉起的独立 PyQt6 热键切换进程无法读取主程序的本地翻译模块，只能通过 Socket 接收 `_get_all_open_trade_windows` 发送的名词。由于该方法原先生成的是纯英文精炼标识（如 `MainConsole`），导致切换框列表里原有的精美 Emoji 和中文名称（如 `💻 主控制台`）全部丢失，变成了简陋的英文标识。

## 2. 实施细节
1. **全局实例绑定**: 
   - 引入 `_global_app_instance` 并绑定到 `StockMonitorApp.__init__()` 初始化生命周期中，供模块级强退函数调用。
2. **构建紧急回收管线 (`emergency_cleanup_subprocesses`)**:
   - 分离提取 `app.qt_process`、`app._hotkey_process`、`app.proc`。
   - 分步实施物理回收: 探测存活 (`is_alive()`) -> 优雅终止 (`terminate()`) -> 短暂等待 (`join(timeout=0.2)`) -> 强制物理抹除 (`kill()`)。
   - 使用 `multiprocessing.active_children()` 全面回收未在主实例中记录的任何后台其他衍生子进程 (如 SyncManager 等)。
   - 给操作系统预留 `time.sleep(0.3)` 缓冲，用以彻底回收管道描述符与系统资源。
3. **注入强退拦截**:
   - **路径 A (`_native_ctrl_handler`)**: 接管 Windows 控制台底层的 3 次 Ctrl+C 连续强制退出分支，注入 `emergency_cleanup_subprocesses()`。
   - **路径 B (CLI Debugger Loop)**: 接管命令行状态下的 KeyboardInterrupt 连续 3 次强退判定，注入 `emergency_cleanup_subprocesses()`。
   - **路径 C (Mainloop Event Loop)**: 保护 Tkinter `mainloop()` 因极速刷新或其他异常捕获到的 Ctrl+C 泄出分支，在 `os._exit(0)` 之前注入 `emergency_cleanup_subprocesses()`。
4. **友好名称根本自愈**:
   - 在 `_get_all_open_trade_windows` 的 `name_map` 构造源头，直接将窗口名装配为高还原的 Emoji 中文友好名称（如 `"💻 主控制台 (MainConsole)"`、`"📊 策略选股与人工复核 (StockSelection)"` 等）。
   - 为概念放量监控子窗口 (MonitorWindow) 在命名时追加 `[MonitorWindow_{win_id}]` 尾缀，在保留精美中文的同时，确保主进程原有的 `"MonitorWindow_" in name` 条件依然能精准过滤出磁贴组 (Tile Windows)，没有破坏分组与键盘滚动联动算法。

## 3. 验证与总结
- 运行 `python -m py_compile instock_MonitorTK.py` 验证语法完全正确。
- 本次修复完美达成了物理安全退场与界面交互的高奢质感统一，让独立进程轮转框无缝恢复了精致的中文 Emoji 排版。
