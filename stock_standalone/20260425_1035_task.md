## 深度优化与系统清理 (Deep System Optimization & Cleanup)
**Date:** 2026-04-25 10:35

### 目标 (Objectives)
对 pyQuant3 全系统进行深度的优化安全分析、性能查验与历史“临时代码/脚本”的深度清理，确保核心业务（竞价、赛马、TK 面板等）安全稳定运行。

### 已完成任务 (Completed Tasks)
- [x] **全项目遗留临时脚本物理清理 (Physical Cleanup of Legacy Temp Scripts)**：
  - 彻底删除了在先前版本迭代中遗留的 `temp_historical_monitor.py`, `temp_old_ver.py`, `tmp_check_db.py`, `tmp_inspect_snapshot.py`, `_inspect_dbs.py`, `_refresh_sensing_bar_snippet.py`, `_refresh_tree_traditional_method.py`, `_repair_signal_db.py`, 和 `analyze_trades_temp.py` 等非业务必须的测试和诊断脚本。
  - 保留了必要的业务闭环修复工具如 `db_repair_tool.py`，保证系统自我治愈能力。
  
- [x] **加固赛马板块 (Sector Bidding Panel) 的线程退出安全性 (Hardened Thread Teardown Safety)**：
  - **问题分析**：此前 `SectorBiddingPanel.closeEvent` 中的资源回收逻辑漏掉了对 `_sbc_thread`（即执行 🧪 SBC 信号后台测试线程）的处理。在执行耗时验证期间若关闭窗口，可能会导致悬挂线程尝试操作被销毁的 UI 组件，引发 C++ 层面的 `RuntimeError` 或内存泄漏。
  - **解决方案**：在 `closeEvent` 中补齐了对 `self._sbc_thread` 的生命周期管控。实现了 `quit()` 和 `wait(2000)` 的平滑退出机制，并提供强力的 `terminate()` 兜底策略，最后调用 `deleteLater()` 彻底清退，保障了竞价面板在任何异常切断状态下的稳定性。

- [x] **全系统临时状态与调试后门排查 (System-Wide Temp-State Audit)**：
  - 深度扫描了 `instock_MonitorTK.py`, `bidding_racing_panel.py`, `signal_dashboard_panel.py` 及 `trade_visualizer_qt6.py`。
  - 确认目前所有的 "测试" 功能（如 "测试买卖策略", "SBC 实时测试"）都是提供给用户的**正规业务特性**（用于回放验证），而非应当删除的临时废弃代码，这些功能的调度流已受控并具有良好的线程安全及容错。
  - 确认 `bidding_momentum_detector.py` 的并发锁保护机制、字典数据防越界及 15:30 收盘持久化逻辑完整健康。

### 结论
系统整体架构经过近期的 `multiprocessing` 与线程锁重构后已基本趋于稳定，临时脚本和过期探索型测试文件均被清除，项目树变得更为纯净和标准化。
