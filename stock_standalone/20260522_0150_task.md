## 🎯 任务背景与核心痛点
用户反馈在实盘或冷启动次日开盘时，竞价赛马板块面板（`SectorBiddingPanel`）在更新后无法显示任何数据，或者在昨日显示全部前只显示一行数据。此Bug属于严重的逻辑死锁，极大影响了交易员早盘黄金时段的行情决策。

---

## 🛠️ 逻辑漏洞诊断与破案过程
经过对数据管道（ Feeder -> Detector -> Worker -> UI Panel ）的深度代码审计，最终定位并根治了两个相辅相成的隐藏逻辑大死锁：

1. **个股评分持久池与板块成员计算门槛过高（Aggressive Filter Gate）**：
   - **痛点**：在 `BiddingMomentumDetector._aggregate_sectors` 板块聚合核心方法中，此前使用了严苛的 `score_threshold`（3.6）用于过滤有效个股和写入活跃个股持久池（`_sector_active_stocks_persistent`）。
   - **后果**：在早盘刚开盘或冷启动阶段，大部分个股的计算分数（例如 1.0、2.0 等）还没能发酵成熟，直接被 3.6 的高门槛彻底过滤并从池中抹除，导致活跃个股持久池在开盘初期直接处于真空期，使板块的各项群体指标（均涨幅、领涨股等）无法被正确累加，最终造成 UI 没有任何板块可以渲染。
   - **修复**：将进入活跃个股持久池与板块合并的阈值统一调低至基础活跃级别 **0.5**。这保证了哪怕个股只有微弱的动量，也能被敏锐地计入板块合力，确保早盘“萌芽阶段”的活跃板块能在第一时间被发掘并呈现。

2. **增量评分收集与空板块白屏的恶性闭环死锁（Incremental Selection Deadlock）**：
   - **痛点**：在 `start_update_scores` 的增量评分更新模式下，系统为了极速性能，只收集处于 `essential`（由已识别的活跃板块龙头与跟随者组成）以及大涨大跌（`pct > 1.5%` 或 `vol_ratio > 2.0`）的股票加入本轮评分。
   - **后果**：当系统冷启动或昨日收盘次日初始化时，活跃板块列表 `active_sectors` 为空，这导致 `essential` 股票池也为空。同时在开盘前期的大部分平稳股票（涨幅在 -1.5% 到 1.5% 之间）因为没有达到阈值，完全不被评分（评分为默认的 0）。这导致它们的分数永远无法突破 0.5 进入持久池，导致 `active_sectors` 无法产生，形成了“没有板块 -> 没有 essential 更新 -> 评分一直为 0 -> 永远算不出板块”的自闭环逻辑死锁！而系统 60 轮（约数十分钟）才进行一次的全量扫描在盘中高频时段完全滞后。
   - **修复**：在增量收集阶段引入**自愈式强制全量扫描**。当系统检测到活跃板块 `self.active_sectors` 长度为空时（即冷启动或开盘初始化白屏状态），强制把本轮设为全量扫描（`scan_all = True`）。这确保了在数秒内算出并呈现出首批活跃板块，瞬间打破死锁。首批板块一旦发掘出来，`active_sectors` 丰富起来，系统又将自动平滑退回到 60 轮一次的高性能增量模式，既消除了白屏，又极大地节约了 CPU。

## 💡 开发原则的工程体现

* **简单至上 (KISS)**：没有通过引入复杂的状态管理或第三方库来解决冷启动，仅通过两行 `if` 自愈判定和降级阈值，以极其微创的手术刀方式直击本质，彻底根治 Bug。
* **精益求精 (YAGNI)**：没有设计所谓的“新版冷启动架构”或大动干戈修改数据总线，完全在已存在的闭环框架内，对原有数据寻址进行对齐与优化。
* **杜绝重复 (DRY)**：保留原有增量/全量扫描的主干逻辑，自愈逻辑在临界区快速插入，完美维护了既有代码的高复用与清洁度。
