# Walkthrough: 解决配置文件监视导致的二次刷新与闪烁问题 (2026-06-25 20:20)

## 1. 痛点分析
1. **二次刷新/闪烁 bug**：用户在策略选股窗口点击表头或切换多级排序时，排序操作会自动调用 `save_ui_states()` 将当前的列宽和排序状态持久化写入 `window_config.json`。
2. **侦听器过度响应**：`GlobalFavoriteManager` 的后台守护线程 `FavoritesWatcher` 每一秒轮询一次，发现 `window_config.json` 的修改时间 (`mtime`) 变动后，会自动调用 `load_from_config()`，而旧版的 `load_from_config()` 在加载完成后会无条件调用 `self.notify_subscribers()`。
3. **连锁反应**：作为订阅者之一的 `StockSelectionWindow` 收到通知后，会将 `_favorites_dirty` 设为 `True`，随后触发 `_refresh_ui_favorites()`。这会导致整个表格的 Treeview 被清空并重新绘制（重新运行 `_do_bulk_render`），即便自选股数据完全没有发生任何改变。这就是用户在点击排序后，界面经历“排序渲染 -> 配置文件更新 -> 监视器检测 -> 触发自选更新 -> 再次全表刷新”两次刷新的根本原因，同时这也导致了选中项被间隔触发选中、闪烁和二次联动的 bug。

## 2. 解决方案设计与实施
### 步骤 I：细粒度脏检查自选变更
在 `global_favorites.py` 的 `GlobalFavoriteManager.load_from_config` 中：
- 重新加载 `window_config.json` 后，在将新加载的自选板块 (`favorite_sectors`) 和自选股 (`favorite_stocks`) 存入内存前，先在加锁保护下与当前的集合进行对比。
- 引入布尔标志 `changed = False`。仅当 `new_sectors != self.favorite_sectors` 或 `new_stocks != self.favorite_stocks` 时，才将 `changed` 置为 `True`，并仅在 `changed` 为 `True` 时才调用 `self.notify_subscribers()`。
- 否则，若仅有界面尺寸、排序等状态更改，仅静默更新 `self._last_config_mtime = mtime`，拦截多余的订阅者通知。

### 步骤 II：编译与测试
- 使用 `py_compile` 对修改后的 `global_favorites.py` 进行物理编译，确认无语法或缩进问题，确保服务安全无缝上线。

## 3. 验收标准
- [x] 点击表头排序或保存窗口尺寸时，不再导致 `GlobalFavoriteManager` 派发多余的通知。
- [x] 策略选股窗口排序时，只执行一次物理渲染，彻底消除二次刷新、界面闪烁与双重联动的现象。
- [x] 编译通过，无任何语法错误。
