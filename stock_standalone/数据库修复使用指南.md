# 数据库修复功能使用指南

## 📋 快速开始

### 基本用法

```bash
# 1. 修复默认数据库 (signal_strategy.db)
python instock_MonitorTK.py -repair-db

# 2. 修复指定的数据库文件
python instock_MonitorTK.py -repair-db your_database.db

# 3. 使用完整路径修复
python instock_MonitorTK.py -repair-db D:\Data\signal_strategy.db
```

## 🎯 使用场景

### 场景 1: 数据库损坏无法打开

**症状:**
- 程序启动时报错 "database disk image is malformed"
- SQLite 错误: "database is locked"
- 数据库文件无法正常访问

**解决方案:**
```bash
python instock_MonitorTK.py -repair-db
```

### 场景 2: WAL 日志未正常合并

**症状:**
- 存在 `signal_strategy.db-wal` 和 `signal_strategy.db-shm` 文件
- 数据库大小异常
- 部分数据丢失

**解决方案:**
```bash
python instock_MonitorTK.py -repair-db signal_strategy.db
```

### 场景 3: 定期维护

**建议:**
- 每周运行一次数据库修复
- 在非交易时间执行
- 保留备份文件至少一周

```bash
# 创建定时任务 (Windows 任务计划程序)
python instock_MonitorTK.py -repair-db
```

## 📊 修复流程详解

修复工具会按以下步骤自动执行:

### 步骤 1: 完整性检查
```
[检查] 完整性检测...
  ✓ 完整性: OK
```
- 如果数据库健康,直接退出
- 如果发现问题,继续下一步

### 步骤 2: 创建备份
```
[备份] 创建备份...
  ✓ 主数据库备份: signal_strategy.db.backup_20260209_122000
  ✓ -wal 备份已创建
  ✓ -shm 备份已创建
```
- 自动备份主数据库文件
- 备份 WAL 和 SHM 文件(如果存在)
- 备份文件名包含时间戳

### 步骤 3: WAL Checkpoint
```
[修复] 尝试 WAL checkpoint...
  ✓ Checkpoint 结果: (0, 2, 2)
```
- 尝试将 WAL 日志合并到主数据库
- 可能解决大部分问题

### 步骤 4: 数据恢复 (如需要)
```
[修复] 尝试 .recover 恢复...
  ✓ 恢复成功
```
- 使用 SQLite 的 .recover 命令
- 从损坏的数据库中提取可用数据

### 步骤 5: 重建数据库 (最后手段)
```
[导出] 尝试读取表数据...
  发现 3 个表: ['follow_queue', 'signal_log', 'config']
  ✓ follow_queue: 150 条记录
  ✓ signal_log: 2340 条记录
  ✓ config: 5 条记录

[重建] 创建新数据库...
  ✓ follow_queue: 已重建 (150 条)
  ✓ signal_log: 已重建 (2340 条)
  ✓ config: 已重建 (5 条)
```

## ⚠️ 注意事项

### 1. 备份文件管理
- ✅ 修复前会自动创建备份
- ✅ 备份文件名格式: `database.db.backup_YYYYMMDD_HHMMSS`
- ⚠️ 定期清理旧备份文件,避免占用过多磁盘空间
- ⚠️ 建议保留最近 3-5 个备份文件

### 2. 数据库锁定
- ⚠️ 修复前请关闭所有使用该数据库的程序
- ⚠️ 如果数据库被锁定,修复会失败
- ✅ 建议在程序未运行时执行修复

### 3. 磁盘空间
- ⚠️ 确保有足够的磁盘空间(至少是数据库大小的 2 倍)
- ✅ 修复过程会创建临时文件和备份文件

### 4. 数据完整性
- ✅ 修复后数据完整性得到保证
- ⚠️ 可能会丢失部分索引和约束信息
- ⚠️ 建议修复后验证关键数据

## 🔍 故障排查

### 问题 1: 提示 "文件不存在"
**原因:** 数据库文件路径不正确

**解决:**
```bash
# 使用绝对路径
python instock_MonitorTK.py -repair-db D:\full\path\to\database.db

# 或者先切换到数据库所在目录
cd D:\MacTools\WorkFile\WorkSpace\pyQuant3\stock_standalone
python instock_MonitorTK.py -repair-db signal_strategy.db
```

### 问题 2: 提示 "数据库被锁定"
**原因:** 其他程序正在使用数据库

**解决:**
1. 关闭主程序 `instock_MonitorTK.exe`
2. 检查任务管理器,结束相关进程
3. 重新运行修复命令

### 问题 3: 修复失败
**原因:** 数据库损坏严重

**解决:**
1. 检查备份文件是否存在
2. 尝试使用备份文件恢复:
   ```bash
   copy signal_strategy.db.backup_20260209_120000 signal_strategy.db
   ```
3. 如果所有方法都失败,考虑重建数据库

### 问题 4: 打包后的 EXE 无法使用
**原因:** 可能是路径问题

**解决:**
```bash
# 使用完整路径
instock_MonitorTK.exe -repair-db D:\Data\signal_strategy.db

# 或在 EXE 所在目录执行
cd D:\path\to\exe
instock_MonitorTK.exe -repair-db signal_strategy.db
```

## 🚀 高级用法

### 在 Python 脚本中调用
```python
from db_repair_tool import DatabaseRepairTool

# 创建修复工具实例
tool = DatabaseRepairTool("signal_strategy.db")

# 执行修复
success = tool.run()

if success:
    print("修复成功!")
else:
    print("修复失败!")
```

### 集成到自动化脚本
```python
import subprocess
import sys

def repair_database(db_path):
    """修复数据库"""
    cmd = [
        sys.executable,
        "instock_MonitorTK.py",
        "-repair-db",
        db_path
    ]
    
    result = subprocess.run(cmd, capture_output=True, text=True)
    
    if result.returncode == 0:
        print(f"✓ {db_path} 修复成功")
        return True
    else:
        print(f"✗ {db_path} 修复失败")
        print(result.stderr)
        return False

# 使用示例
repair_database("signal_strategy.db")
```
