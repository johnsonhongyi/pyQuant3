# Task: 修复竞价赛马面板(BiddingRacingRhythmPanel)退出异常、PyEval_RestoreThread 崩溃与 I/O 卡顿 (统一存盘与延迟引用释放版)
日期时间: 2026-05-28 10:55

## 1. 目标与痛点
- **痛点 A (析构期 GIL 释放与线程销毁崩溃)**: 在关闭 `BiddingRacingRhythmPanel` 时，由于非 GUI 线程在进行数据回调时，面板的 `closeEvent` 会执行 `QApplication.processEvents()`。这在 Nuitka 虚拟机环境下，极易在销毁过程中引发多线程重新竞争 GIL，导致 `PyEval_RestoreThread: the function must be called with the GIL held` 致命错误。
- **痛点 B (二次启动后关闭的 GIL 崩溃)**: 当用户第二次打开赛马面板并将其关闭时，主 Tkinter 窗口接收到 Qt 同步发出的 `closed` 信号后，其回调 `_on_racing_panel_closed` 会立即执行 `self._racing_panel_win = None` 清空强引用。若此时 C++ 的 `closeEvent` 流程尚未完全结束，由于强引用被立即置空导致 Python 包装器对象在 C++ 销毁完成前提前被 GC 垃圾回收，从而再次触发 `PyEval_RestoreThread` 的 Access Violation 崩溃闪退。
- **痛点 C (退出同步 I/O 卡顿)**: 关闭面板时，主面板会遍历并 `close()` 每个子对话框。这会触发子窗的 `closeEvent` 分别同步执行 `_save_header_state` 写盘，造成多次重复的 `gzip.write()` 磁盘落盘动作，严重阻塞 GUI 主线程，导致退出时卡顿 1-3 秒。如果直接屏蔽写盘，又会导致子窗口最后调整的宽度与几何尺寸在退出时丢失。
- **目标**: 彻底重构 `BiddingRacingRhythmPanel` 和子对话框的 `closeEvent` 清理顺序，安全撤销重入式事件处理，规避退出时的冗余物理写盘，同时将窗口强引用清空操作安全移出 C++ 退出执行帧，实现零延迟、零配置丢失、纯异步安全的对象生命周期回收与销毁。

## 2. 实施细节
1. **移除 unsafe `processEvents` 强制泵送**:
   - 彻底删除 `closeEvent` 中的 `QApplication.processEvents()` 调用，以物理上阻断在清理与销毁阶段所产生的高危重入和 GIL 竞争。
2. **延迟清除 PyQt 窗口强引用**:
   - 将主 Tkinter 窗口中的 `self._racing_panel_win = None` 强引用置空与同步窗口状态操作移出同步信号回调，重构为委派给 Tkinter 的 `self.after(100, _safe_clear)`。
   - 这样能保证 `self.deleteLater()` 在 Qt 事件循环中安全排队、`closeEvent` 彻底退出且 C++ 状态完全稳定后，才会在 Tkinter 的下一帧安全地清除 Python 强引用并触发垃圾回收，彻底根除了析构期间的 GIL 争抢崩溃。
3. **重构子对话框安全回收逻辑**:
   - 在主面板关闭前，显式执行 `disconnect()` 阻断子对话框对 `data_updated` 的信号订阅，彻底根除销毁过程中子对话框接受后台残留回调导致向空指针/死对象渲染的崩溃。
   - 对 `pie_widget` 内部的动画刷新 QTimer 定时器和子窗口直接调用 `.stop()` 后再安全调用 `deleteLater()`。
4. **注入 `_is_main_closing` 标志并由主面板统一保存位置与列宽**:
   - 在主面板遍历关闭子窗前，注入 `child._is_main_closing = True`。子窗口的 `_save_header_state` 检测到该标志后直接 `return`，完全跳过子窗口各自的磁盘 Gzip 写入，彻底消除了退出卡顿。
   - 为了防止位置与列宽数据丢失，在主面板的 `_save_ui_state` 中引入 `sub_configs` 收集器。遍历活跃的 `_detail_dialogs` 子窗口，实时读取它们当前的几何尺寸与表格列宽，生成对应的 `"detail_column_widths"`、`"detail_geometry"` 或 `"cat_detail_widths_xxx"` 配置项，并一同更新合并到主配置字典中，由主面板发起唯一一次原子物理写盘。
5. **调整销毁通知时序**:
   - 在 `super().closeEvent(event)` 成功返回之后，再执行 `self.closed.emit()` 广播，确保父进程 `MonitorTK` 能够获知“该面板已经彻底准备好回收其 Python 与 C++ 实例”，避免在实例析构一半时触发主进程访问冲突。
6. **添加 deleteLater 物理延迟销毁机制**:
   - 移除不安全的同步销毁手段，将所有需要回收的对象全部通过 `deleteLater()` 安全地放回到 Qt 顶层事件循环进行物理延迟销毁，从而保障 Windows OS 在垃圾回收时不会发生底层冲突。
