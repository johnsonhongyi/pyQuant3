# 2026-05-22 12:50 板块聚合极限性能重塑任务清单 (Resolving Bidding Momentum Detector Aggregation Lags)

## 1. 痛点定位与分析

### 🔴 核心痛点
在 `BiddingMomentumDetector` 计算的收尾阶段，`_finish_score` 会同步触发 `_aggregate_sectors` 进行板块数据深度聚合。虽然打分计算本身已经被重构为平滑的分片状态机调度器，但是 `_aggregate_sectors` 内部存在两大计算瓶颈，在纯 Python 层面会产生高负荷的密集计算循环：
1. **联动板块跟风率的嵌套大循环**：每个龙头都会有很多关联概念，系统对每个概念下的几十个成员又进行了一次全量遍历计算。在板块很多时，产生了致命的 $O(K \times C \times N)$ 运算复杂度，导致主线程在后台线程霸占 GIL 期间被饿死（GIL Contention & UI Starvation），产生 3-5 秒的系统假死与卡顿。
2. **逻辑顺序倒置与冗余构造**：在板块进行门槛校验（`board_score < threshold`）过滤之前，系统就提早对所有板块的所有强势股进行了大量的角色判定、形态暗示获取、`race_candidates` 复杂字典数组的构造以及 `all_member_codes` 全体跟风遍历。这使得 90% 以上注定被过滤剔除的弱势板块白白消耗了成百上千次的 CPU 运算与 GC 内存开销。

---

## 2. 解决方案与重构策略

我们采取了极客化的**“两阶段过滤 + 单次跟风缓存”**的组合拳来重塑该核心算法，实现了物理级零冗余开销：

### 🛠️ 核心优化一：两阶段初筛与过滤门槛校验 (Two-Stage Early-Exit Filtering)
- **第一阶段（极速初筛判定）**：在板块循环入口，仅快速计算 `follow_ratio`、`avg_pct` 以及板块总分 `board_score` 等基本盘信息，并直接进行初筛和门槛校验（门槛下调与入榜校验）。如果板块无法进入榜单，直接在第一时间调用 `continue` 退出，**实现 0.00ms 物理零开销**！
- **第二阶段（高昂包装延迟加载）**：只有当板块确认能够保留并登入榜单时，才在后半段执行高昂的个股角色判定、形态获取、`race_candidates` 详细字典包装以及 `followers` 详情字典的重新拼装。这一简单的顺序调整直接把大循环的无用负载砍掉了 **90% 以上**！

### 🛠️ 核心优化二：Single-pass Concept Cache 联动概念本轮缓存与门槛收紧
- **引入单次聚合本轮缓存**：在 `_aggregate_sectors` 顶部声明一个局部缓存字典 `concept_cache = {}`。在计算每个龙头的概念跟风率时，如果同一概念在同一次聚合计算中已经算过，直接 O(1) 常数级复用缓存，非活跃或不合要求概念直接记为 `None` 短路，将原本 $O(N^2)$ 嵌套循环的计算吞吐量直接缩减了 **95% 以上**！
- **联动门槛收紧**：仅当板块强势有活力（`board_score >= 5.0` 或 龙头涨幅 `pct >= 1.0`）时才运行 `linked_concepts` 联动概念跟风判定。弱势板块直接跳过该操作，进一步稀释了无用算力开销。
